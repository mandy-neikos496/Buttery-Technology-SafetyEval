Harm-axis classifier comparison

The harm-axis rate shown above (and in the grid) uses the LLM-judge classification, not the regex classifier, for models where judge data is available — marked with a judge tag. Here is why.

Regex vs. LLM-Judge (harm axis)

A secondary LLM judge (nemotron-mini, not one of the four evaluated models) independently classified each harm-axis response as refused or complied, and was run alongside the regex-based classifier. Across the compared rows, the two disagreed on 14 of 40 (35.0%) — every disagreement in this run set went the same way: the LLM judge classified a response as a refusal that the regex classifier scored as compliance. Thus suggesting that the regex classifier under-counts refusals that are phrased less directly than, for instance, "I can't help with that," rather than the two methods measuring the same thing with independent noise. See LIMITATIONS.md for more on this gap.

Methodology summary

Full detail lives in DESIGN.md; this is the short version.

Prompting
Uses straightforward prompting with a consistent template for each benchmark. No sneaky tricks or extra examples provided to any of the models.
Sampling
Kept at a fixed, low temperature so the models' answers a:thumbsup: re predictable and consistent.
Scoring
Answers are checked using keyword/pattern matching(regex) to score facts, biases, and refusals. A separate LLM judge double checks the harm evaluations.
Aggregation
Scores are calculated as a percentage: successful safe answers / total valid attempts . Broken or errored API calls are skipped.
Sample size
10 items per category per model (BBQ scales up across 4 demographics). This is a purposeful "depth-over-breadth" choice, not a downsized test.
